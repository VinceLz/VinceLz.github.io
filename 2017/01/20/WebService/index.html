WebService

Veröffentlicht am 2017-01-20

#前言#
WebService ?

Service听过，但是WebService是什么鬼东西呢？难道说时web端的服务吗？web的服务一般不是都会
请求后端接口吗？

Web service是一个平台独立的，低耦合的，自包含的、基于可编程的web的应用程序，可使用开放的XML（标准通用标记语言下的一个子集）标准来描述、发布、发现、协调和配置这些应用程序，用于开发分布式的互操作的应用程序。[1]
Web Service技术， 能使得运行在不同机器上的不同应用无须借助附加的、专门的第三方软件或硬件， 就可相互交换数据或集成。依据Web Service规范实施的应用之间， 无论它们所使用的语言、 平台或内部协议是什么， 都可以相互交换数据。Web Service是自描述、 自包含的可用网络模块， 可以执行具体的业务功能。Web Service也很容易部署， 因为它们基于一些常规的产业标准以及已有的一些技术，诸如标准通用标记语言下的子集XML、HTTP。Web Service减少了应用接口的花费。Web Service为整个企业甚至多个组织之间的业务流程的集成提供了一个通用机制

这是我从百度抓取的一些文字，根据文字的内容，我们得知他是基于Socket技术的，是低耦合，用户分布式的应用程序，它可以使
通用的机制。 根据这些内容，我们大概就是知道，这货写的东西其他语言，平台都可以使用，没有障碍，独立的一种服务技术。

#Socket#
Socket 大家应该都知道吧

Socket 分为客户端/服务端，基于这种通信一次就会断开，所以我们服务端就必须跑一个while循环一直来监听对应端口来实现服务。
那么socket有什么特点呢？ 需要开启端口，长连接通信，也容易被防火墙拦截，所以导致有时候我们的连接会被防火墙杀掉，导致我们没有接受到
任何消息，所以一般的趋势就是采用 “心跳检测” 每过一定的时间，客户端/服务器就会发生一个数据包，以此来保持连接不被断开。

#http#
http 这是基于web服务的协议，这种协议的底层还是Socket ，只不过它对socket进行了二次封装
http特点：跨平台比较好，传输数据比较麻烦。get/post

#WebService#
比较http /socket 我们发现都不是很友好，我们需要寻找一种替代方案，那就是我们今天要讲的WebService,他能使的运行在不同
机器上的不同应用无须借助附加的软件、硬件，就可以很好的数据交互。

#Java wsImport#
wsImport 是Java为我们提供的一款引入WebService命令软件

使用如下：

wsImport [option] <url>

-d 指定要输出的目录  

-s表示要解析java代码，默认解析的是class字节码

-p 指定要输出的包

其实就是这么简单，我们去找别人发布的WebService的地址，然后我们使用wsImport就可以生成一个jar包，jar包里面就包含了别人的服务，那就有
人要问了，这不就是java 的一个jar包吗，我可以自己写完发布呀!,说是这样的，但是你的jar包可以给c# 用吗？不可以吧，你还需要为其他语言写一套
代码去，但是如果你采用了WebService的话，只需要一套代码，天下任你行。

#如何发布#
java发布其实很简单，

@WebService   // 声明该业务类对外提供WebService服务
public class demoService{

        public String getPhone(String os){

          if(os.equal("ios"))
          return "iPhone8"
          else if(os.equal("android"))
          return "android手机"
        }
      public static void main(String []args){

        String addr="http://127.0.0.1:8080/ws/phoneService"
        Endpoint.publish(address,new demoService())
      }

}

就这样，我们就向外部发布了一个服务了，别人只要拿着http://127.0.0.1../.....这个url就可以使用我们的服务了，是不是很简单，而且我们服务器的一个端口就
可以发布多个服务，只要保持服务名字不相同即可

其实当我们游览器输入http://127.0.0.1:8080/ws/phoneService呢？他会显示出一些xml文档，没错，WebService服务的发布就是基于xml文档结构说明的，

看见没在xml文档中详细说明了我们服务的结构特征，他详细描述了我们方法的返回值。参数，内容等信息，并且这些信息我们可以定制

@WebMethod()

@WebResult()

@WebParam()

使用这3个注解，就可以指定我们想要的名字。

#WebService优势何在#

可以作为接口的3种方式

  1. socket
  2. http
  3. WebService

但是前2种只能进行简单的字符串传递，对于一个对象呢？没错，WebService可以很少的解决这个问题，他实现了夸语言，夸平台的优势

#四个概念#

  1. SOAP 简单的对象访问协议(http+xml)
  2. SOA 面向服务的架构，他是一种思想,IBM大力倡导
  3. WSDL WebService描述语言
  4. uddi 目标服务，统一描述，发现，继承

以上都是我们基于jdk为我们提供的工具，但是我们想和Tomcat服务器结合起来，WebService会随着spring 同时启动，如何做呢，这就需要
我们使用框架来继承了

#框架#

CXF介绍：
    cxf是 celtrix和xfire合并而成的，并且捐赠给apache
    CXF的核心是org.apache.cxf.Bus
    CXF默认是依赖于spring 的

CXF特点：
    于spring servlet 做到了无缝连接，
    支持注解形式来发服务
    能够显示一个WebService服务列表
    能够添加拦截器，输出、输入、日志、用户权限等

#如何使用#

第二种方式为JaxWsServerFactoryBean 来发布服务

  1. 是ServiceFactoryBean的子类，也是功能上的扩展
  2. 此类，必须在服务类上假如@WebService注解，如果不加，不会报错，但是也不会暴露接口
  3. 可以通过拦截器对请求进拦截
