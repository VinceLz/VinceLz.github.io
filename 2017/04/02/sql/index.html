优惠劵的实现

常见的优惠券发放途径有两种：

  • 1活动统一发放：给不活跃用户短信通知发放优惠券来促进活跃、提高留存，给特定活动造势：女神节全体女性用户优惠券等。
  • 2用户操作触发：活动页面发放，订单满额发放，邀请好友发放，新用户注册发放等。

一般的优惠劵触发的场景就这2种，所有我们的项目目前只考虑用户操作触发,也就是当用户注册&购买金额到达一定的区间就会放发对应的优惠劵。

优惠劵的使用规则：
优惠券的实质就是通过“价格歧视”来让用户感到享有特权，从而完成产品想实现的效果，具体“歧视”手段也有很多：

  • 1用户：新用户专用，女性用户专用，黄金用户专用，北京地区专用
  • 2商品：电子产品专用，商家专用
  • 订单：满额使用，在线支付专用

优惠劵状态
优惠券状态

  • 1待使用：发放后的正常状态，可以正常使用；
  • 2占用中：订单使用优惠券但是未付款，优惠券占用中不可使用，根据订单后续状态转化状态，订单失效转为待使用。高并发下很有必要
  • 3已使用：订单支付成功后标记优惠券已使用。
  • 4已失效：优惠券超期未使用失效。

优惠劵的遗留问题

完成了优惠劵的业务处理，这里有的问题，就是优惠劵有过期日期,那么我当时在设计的时候使用了status 2表示该卷过期了，那么很简单，我们需要跑一个调度系统，每晚0：00对数据库进行一次遍历即可，找如果有日期过期，则让status =2有以下方案:

  • 1 spring & Java 的调度任务
  • 2 linux crontab 调度任务
  • 3 MySQL自带的时间调度

最终我采用了mysql自带调度，性能肯定要比上述2种快。
